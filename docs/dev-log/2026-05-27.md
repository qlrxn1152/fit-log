## 발생한 문제

### 1. 문제 제목
#### 상황
- MemberRepository 테스트 코드를 작성하면서 No ParameterResolver Registered Error 

-ParameterResolutionException: No ParameterResolver registered for parameter [. . .] in constructor


#### 원인

```
@RequiredArgsConstructor
class MemberRepositoryImplTest {

    private final MemberRepository memberRepository;

```

이 코드는 오류수정전 테스트코드.

Test 는 현재, 스프링에 의존하지 않고 순수 JUnit 객체로만 작성된 상태.

애노테이션은 @RequriedArgsConstructor => 롬복은 아래와 같이 생성자를 만들어줌.

```
public MemberRepositoryImplTest(MemberRepository memberRepository) {
    this.memberRepository = memberRepository;
}
```


이후에 스프링이 생성자 1개인것을 확인하고 @Autowired 로 자동주입 시도
```
@Autowired
public MemberRepositoryImplTest(MemberRepository memberRepository) {
    this.memberRepository = memberRepository;
}
```



하지만, JUnit 입장에서는 스프링에 의존하지 않으므로, MemberRepository 를 어디서 가져와서 이 객체를 주입해줄지 전혀 모르는상태였음 즉, JUnit이 생성자 파라미터를 해결하지 못함. 

그래서 에러메세지로 나온 메세지가 

```
-ParameterResolutionException: No ParameterResolver registered for parameter [. . .] in constructor
```

즉, Test => Junit 만 실행이됐던거고, 그래서 스프링 컨테이너가 뜨지 않았고, 그래서 @Component 로 등록한 객체를 주입받을수 없는상태 또한, @Autowired 도 작동되지 않는 순수 Junit 으로 작성한 테스트코드

그래서, 

```
@Repository
@Slf4j
public class MemberRepositoryImpl implements MemberRepository {
```

테스트코드가아닌 실제 객체에서 이렇게 @Repository 를 등록했음에도 ( @Repository -> @Component 포함 ), 테스트코드에서 돌렸을때 에러가 나왔던거.

그럼 이를 어떻게 해결? 

- 순수 단위 테스트 코드를 작성
- 스프링 주입


---

#### 해결

save, FindById, FindAll 을 하는데에 굳이 Spring 을 안써도 상관없으니 순수 JUnit 으로 하기로했음.

```
class MemberRepositoryImplTest {

    private final MemberRepository memberRepository = new MemberRepositoryImpl();
```

그래서, 객체를 직접 주입해줌

---

#### 배운 점

-Test 코드는 일단 기본적으로 Spring 에 의존하지않는상태.

그래서 스프링이 필요하다면 직접 @SpringBootTest 애노테이션을 넣어줘야함.

## 발생한 문제

### 2. 문제 제목

#### 상황
- 로그인상황에서, 유저가 서버로 보내온 데이터를 @ModelAttribute 로 값을 받았는데 컨트롤러에서 값을 받아오지못함 ( null )

---


#### 원인

-DTO 에 Setter 주입자를 안열어줬었음.

---

#### 해결

```
@Getter
@ToString
public class LoginForm {

    @NotBlank(message = "아이디를 입력하세요.")
    private String loginId;

    @NotBlank(message = "비밀번호를 입력하세요.")
    private String password;

}
```
우선은 오류를 수정하기전, 로그인 전용 Membmer DTO 이다.

```
@Getter
@ToString
@EqualsAndHashCode
@Slf4j
public class Member {

    private Long id;
    private String loginId;
    private String password;
    private Grade grade;

    public Member(String loginId, String password) {
        this.loginId = loginId;
        this.password = password;
        this.grade = Grade.USER;
    }

    public void setId(Long id) {
        this.id = id;
    }
```

위는 실제 사용하는 도메인 Member.

예전에, Setter 주입자는 되도록이면 안쓰는게 좋다하여, DTO 를 사용할때도 Setter 를 사용하지 않으려고했음.

Setter 주입자를 사용하지않고, @ModelAttribute 를 사용하려고하니, 클라이언트가 보낸 데이터를 LoginForm 객체에 넣어주지 못하고있었고, 그래서 null 로 출력이됨.

@ModelAttribute 는, Setter 를 통해서 값을 주입해주는데 Setter 가 열려있지않으니, 데이터를 넣어줄수없던상황.

결국, @ModelAttribute 는 @RequestParam -> setXxx 를 통해서 값을 주입해주는 기능 

그래서, Setter 주입자를 열어줘야 null 이 아닌 실제 보내온 데이터로 입력 

---

#### 배운 점

- Setter 주입자는 아예 쓰면 안되는줄알았는데, DTO ... 요청 데이터를 담아두는 데이터에는 Setter 를 사용하는게 일반적이라고 한다.

다만, 실제 도메인 객체 ( Member ) 에는 Setter 를 되도록이면 사용하지 않도록 해야한다. 차라리 생성자 / 메서드를 사용해서 값을 변경하는게 더 좋다고함 

한줄로 정리해보면, 
```
DTO/Form 객체: @Getter @Setter 허용
Domain 객체: @Getter 중심, 생성자와 의미 있는 변경 메서드 사용
```
---


#### 별도로, 왜 도메인에서는 Setter 를 권장하지 않고, DTO 는 괜찮다는거지 ? 

- 도메인 객체에서는 객체의 상태ㅔ가 아무곳에서나, 아무 의미 없이 변경될 수 있기 때문이라고함 ( 명확성 ? )

그럼 DTO 에서는 이러한 것들이 괜찮은건가? DTO 는 그럼 아무데서나 변경되고 해도 괜찮다는건가 ? 

우선 도메인 객체에서 Setter 주입자를 열어서 아래와 같은 상황이 발생했다고 가정해보자.

```
member.setLoginId("changed");
member.setPassword("1234123");
member.setName("asdfzvc");
member.setId(999L);
```

어떤곳에서 이러한 코드들이 막 사용 되어져 있다고하자.

1번 파일에서 로그인아이디설정하고, 패스워드설정하고 / 3번파일에서도 바꾸고, 6번파일에서도 name,id 를 바꾼다고하자.

그럼, 이 코드를 짠 사람이 아닌 다른사람이 setPassword 같은 코드를 봤다고해보자

그럼 이 사람 입장에서는 이 코드가 지금 

초기 비밀번호 설정인지 ( 회원가입할때 설정하는건지 ) / 비밀번호를 변경하는건지 ... => 의도가 명확하지 않게 보일수도있음 

그래서, 도메인에서는 Setter 를 사용하지않고, 의미있는 메서드를 뽑아서 사용함 

```
    public void changePassword(String encodedPassword) {
        if (encodedPassword == null || encodedPassword.isBlank()) {
            throw new IllegalArgumentException("비밀번호는 비어 있을 수 없습니다.");
        }

        this.password = encodedPassword;
    }
```

이런식으로 메서드를 뽑아서 이를 사용하는 입장이면, 
```
member.changePassword("changePassword")
```

이름만으로도 '아 비밀번호를 변경하는구나' 라고 생각할수있음.

그럼 이 메서드를 직접 타고 들어가서 확인해봤을때, 해쉬로 저장이 되어지는구나도 알수있음 

즉, 이런 명확성과 같은것때문에 도메인에서는 setter 주입자보다 따로 메서드를 뽑아서 사용하는것을 적극 권장.

그럼, DTO 에서도 메서드를 뽑는게 더 좋지않나? DTO 에서는 왜 Setter 괜찮은거임 ? 

- DTO 는 도메인이아닌, 데이터를 전달하는데에 의의가있음

위 예제를 확인해봤을때, 

-Member / LoginForm / 유저가 서버로 보내온 LoginId, Password 

즉, DTO ( LoginForm ) 은 유저가 서버로 보내온 데이터들을 Member 도메인으로 전달하는 역할 ... 

그래서 이럴때에 잠깐 쓰는 객체임 => 그래서, Setter 를 사용하는 실용성이 더 큼 ( 다만 DTO 에서 Setter 주입자의 문제가 생기지 않는것은 x)

## 오늘 배운 점 정리

- Domain -> Getter / 생성자 / 메서드 위주
- DTO -> Setter 허용 

- Test : JUnit 객체 ... / 스프링에 의존x => 스프링이 필요하면 @SpringBootTest 사용 
- @ModelAttribute => @RequestParam -> Setter 주입자로 값을 주입
